# `--branch` errors on an in-use branch rather than falling back to a Derived Branch

The `--branch` CLI flag and the interactive From Branch entry now share one resolution: exists (local or remote) → Direct Checkout, doesn't exist → New Branch off the Base Branch. This replaced the old `--new-branch` flag, whose "existing branch" case was already broken (`git worktree add -b` fails on a name that already exists), and fixed a latent bug where `--branch` only checked local branches, so a remote-only branch name could spawn a diverging local duplicate instead of checking it out.

This means `--branch main` still fails in the common case, since `main` is almost always already checked out in the primary worktree. The alternative — silently falling back to a Derived Branch (as the quick-pick `main`/`master` menu option does) — was rejected: `--branch` is an explicit request for a specific branch, and silently reinterpreting it into a different branch would be surprising, especially in non-interactive/CI use where there's no prompt to catch the substitution. Anyone who wants a Derived Branch off `main` should use the interactive quick-pick, which is designed for that case.
